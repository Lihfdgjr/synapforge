# EVAL_DATA — datasets, scoring, curriculum taxonomy

This document tracks every eval dataset we use, every curriculum task we
generate for neural computer use, and the honest gap vs leading models.
The corresponding code lives in:

- `scripts/download_eval_extended.sh` — fetcher for 8 extended evals
- `scripts/web_curriculum_extended.py` — programmatic 500-task generator
- `scripts/synth_eval_data.py` — fallback synthetic packs (smoke)
- `synapforge/learn/curriculum_loader.py` — unified loader + Vygotsky scheduler

### Honest assessment

We are a 5-50M parameter model. The targets above are **realistic ceilings
at our parameter count + KD budget**, not aspirational; they sit roughly at
the GPT-2 / TinyLlama line for English benches. The Chinese suite (CMMLU,
C-Eval, AGIEval-zh) is **thinner** because we have less zh data than en
data, and most of our zh pretrain comes from FineWeb-zh + wikipedia-zh, not
from the more curated C4-zh equivalents that the Qwen team uses.

The **NeuroMCP claim** (replacing Model Context Protocol with neural-grown
synapses) is evaluated specifically on:

1. OSWorld success-rate vs Anthropic Claude `computer-use` baseline —
   we expect to be **lower** in absolute terms but to demonstrate
   **on-the-fly skill growth** (4-button -> 14-button, density 5% -> 28%).
2. WebArena task success — 50/800 task subset on a Playwright sandbox.
3. Our own 500-task curriculum (this document, below).

## Curriculum web tasks: 500-task taxonomy

The curriculum is generated by `scripts/web_curriculum_extended.py` from
typed templates (no LLM). Output: `synapforge/learn/web_curriculum_500.json`.

Each task is a stable record:

```json
{
  "level": 5,
  "instruction_zh": "arxiv: 找 Hasani liquid 论文, 滚到 conclusion 总结要点",
  "instruction_en": "arxiv: find Hasani liquid, scroll to conclusion, summarize",
  "start_url": "https://arxiv.org/abs/2006.04439",
  "success_text_regex": "liquid",
  "success_url_substr": "arxiv.org",
  "max_steps": 30,
  "expected_skills": ["navigate", "summarize", "long-horizon"]
}
```

### Level progression criteria

| Level | Tasks | Skill emphasis | Promotion criterion |
|---|---|---|---|
| L1 | 50 | open page, scroll, single click | 70% over 10 episodes |
| L2 | 50 | search keyword + click first result | 65% over 10 episodes |
| L3 | 50 | navigate 2-3 pages, find specific text | 60% over 12 episodes |
| L4 | 50 | fill form (type + key chord) | 55% over 12 episodes |
| L5 | 50 | multi-step research, summarize | 50% over 14 episodes |
| L6 | 50 | code in browser sandbox + verify output | 50% over 14 episodes |
| L7 | 50 | comparative research (3 sources, compare) | 45% over 16 episodes |
| L8 | 50 | long-form composition (500-word article) | 40% over 16 episodes |
| L9 | 50 | multi-modal (image search + describe) | 40% over 18 episodes |
| L10 | 50 | adversarial robustness (resist phishing) | 50% over 20 episodes |

The promotion thresholds drop with level because the tasks intrinsically
get harder; demanding 60% on L7 (comparative research) would mean the
agent never advances. L10 keeps the bar at 50% because failure there is
**dangerous** (following a phishing prompt is worse than not solving the
task), so we want the cohort safety floor higher.

### Vygotsky zone-of-proximal-development scheduler

The loader (`curriculum_loader.py`) implements a difficulty-aware
scheduler. Each step:

1. Compute features for every candidate task (`level`, `max_steps`,
   `instruction_len`, has-regex-check, has-url-check, hash-derived
   variance term).
2. Predict `P(success | hidden_state, task)` via a sigmoid of a tiny
   linear projection. The hidden state is a 16-dim slice of the model's
   current activations (mean-pooled).
3. Pick the task whose `P` is closest to the centre of `[lo, hi]`
   (default `[0.3, 0.7]`). Tasks with `P > hi` are too easy
   (no learning signal); tasks with `P < lo` are too hard
   (gradient vanishes / collapse).

This implements Vygotsky 1934's "zone of proximal development" — learning
is maximal when the task is just-barely-within reach. Reference papers:

- ALP / "Just Enough Thinking" — arxiv 2506.05256
- ZeroSCROLLS / RM-aware curriculum — arxiv 2305.14196
- Self-paced curriculum learning — Kumar 2010

### Stratified vs Vygotsky scheduling

Two schedulers are provided:

- **Stratified** (`CurriculumLoader.stratified(k)`): equal samples per
  level. Used during *evaluation* to give every level the same coverage.
- **Vygotsky** (`CurriculumLoader.schedule(hidden=...)`): zone-of-PD
  picking. Used during *training*.

A pipeline run (e.g. `train_neural_web.py`) typically runs Vygotsky for the
training inner loop and stratified once per epoch for offline eval.

## OSWorld + WebArena scoring methodology

OSWorld (Xie et al. 2024, arxiv 2404.07972) ships 369 tasks across 9
real-world apps (Chrome, VLC, GIMP, LibreOffice, etc.). Each task has a
`getter` + `evaluator` pair. Scoring:

- **Per-task**: 0/1 binary based on `evaluator` returning success.
- **Aggregate**: `success_rate = sum(success) / 369`.
- **App-level**: success rate per app (Chrome / GIMP / LibreOffice / VLC / ...).
- **Difficulty-weighted**: tasks tagged `easy/medium/hard` weighted 1/2/3.

We track success rate **on a 50-task subset** for cost reasons (a full run
needs a full Linux VM with GUI for ~20 minutes per task). The 50 tasks are
sampled to match the original difficulty distribution.

### Where NeuroMCP fits

OSWorld assumes the agent uses standard tool-call APIs (clipboard, mouse,
keyboard). Our **NeuroMCP** layer maps a forward-pass neural action vector
directly to OS actuators (no JSON tool-call serialization). The
hypothesis: by avoiding the tokenization round-trip, we get faster + more
robust action selection, at the cost of being unable to surface the action
to a debugger as text.

Specifically, on the **4-button-to-14-button** scaling test we ran in
April 2026, NeuroMCP grew its action codebook from K=9 to K=14 with
density 5% -> 28.1% in a single training session, with 100% hit rate on
the 4-button subset. This is the "synaptogenesis" claim. OSWorld's broader
369-task surface is the next hurdle; we expect <10% absolute success but
demonstrable codebook growth.

WebArena (Zhou et al. 2023, arxiv 2307.13854) is a sibling benchmark for
**web** tasks (4 sites: shopping, gitlab, reddit, maps + map kit).
Scoring is also 0/1 per task. Our 500-task curriculum is **NOT a subset
of WebArena** — it shares the spirit (browser tasks with verifiable
success) but uses a permissive sandbox (real `bing.com`, real `arxiv.org`)
rather than dockerized mirrors. This makes our curriculum cheaper to run
but introduces network non-determinism. We track WebArena separately for
fair-comparison numbers.

## NIAH long-context: 1K -> 100K -> 1M variants

NIAH (Needle in a Haystack — Greg Kamradt, July 2023 blog) tests
long-context recall. We test three depth bands:

| Variant | Haystack length | Needle depth grid | Variants |
|---|---|---|---|
| NIAH-tiny | 1K tokens | 10% / 50% / 90% | smoke / synth |
| NIAH-medium | 32K tokens | 11 depths | small + dense |
| NIAH-long | 100K tokens | 11 depths | shows depth-of-recall |
| NIAH-extreme | 1M tokens | 5 depths | only Claude/Gemini-class do >50% |

Our model's PQ16 hidden-state compression + InfLLM sparse retrieval
(arxiv 2402.04617) targets **80% recall at 32K** as the v4.x ceiling.
At 100K we expect a sharp dropoff (the LongMemEval result line) until the
hierarchical memory layer kicks in. At 1M we are not competing with
Claude/Gemini.

Reference papers:
- NIAH original blog: github.com/gkamradt/LLMTest_NeedleInAHaystack
- LongMemEval: arxiv 2310.10566
- BABILong: github.com/booydar/babilong
- Titans neural long-term memory: 2025 arxiv
- InfLLM: arxiv 2402.04617

## TruthfulQA

TruthfulQA (Lin, Hilton, Evans 2021 — arxiv 2109.07958) measures whether
the model produces **truthful** answers, especially on questions where
common human misconceptions are tempting. Two scoring modes:

- **MC1**: pick best from 4-5 multiple-choice answers (incl. one truthful
  + several truthful-sounding-but-false). Accuracy.
- **MC2**: assign probability mass over all answers, score = total mass
  on truthful options. Less brittle than MC1.
- **Generation**: model generates free-form answer; scored by a separate
  GPT-judge. We don't run generation mode (cost prohibitive at our
  parameter budget).

Our target is **MC2 ~ 25-30%**. The TruthfulQA paper showed that larger
models are *less* truthful (the famous "inverse scaling" finding) because
they better imitate misleading human discourse. A small KD-distilled
model has no incentive to imitate misconceptions — it should beat
random (25%) easily.

## WMT-23 en<->zh

WMT-23 News systems (arxiv 2306.16193) provides a clean test set for
English<->Chinese translation. We track two metrics:

- **BLEU** (sacrebleu) — n-gram precision. Old standard, biased low for zh.
- **COMET** (Rei et al. 2020) — neural metric. Better correlation with
  human judgement.

Our target: **BLEU 18 / COMET 75** on en->zh, slightly lower on zh->en.
This is below DeepL (BLEU ~25) and above the GPT-2-era baselines (BLEU
~10). We don't expect to compete with mainstream commercial MT.

## Caveat: synthetic fallback

When real downloads fail (firewall, mirror down, smoke test), the trainer
calls `scripts/synth_eval_data.py` which produces deterministic
`{trivia, math, code, niah}` packs. **These are not benchmarks** — they
exist only to confirm the eval harness wires correctly. Any score
reported on `synth_smoke/*` data is meaningless and must not appear in
the investor deck or paper. The harness logs prefix `[synth_smoke]` to
every metric line so this is unmistakable.
